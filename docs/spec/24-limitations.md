# 24. Limitations

Honest accounting of what this design does not solve, trades away, or leaves genuinely uncertain â€” a spec that only lists strengths has not been stress-tested.

## 24.1 Zero production track record

Every "genuinely superior" row conceded in Â§22.1 is a real cost: LangGraph's checkpointer has years of production edge cases behind it (including the bugs documented in Â§2.3 â€” bugs are evidence of real usage, not just weakness); Ulexite's checkpoint/replay design (Â§10.4, Â§18) is unvalidated against production failure modes (partial writes during a crash mid-checkpoint, clock skew across a distributed provider registry, adversarial trace-log tampering) that only surface at scale. The spec asserts these are solvable within the architecture; it does not claim they are solved.

## 24.2 The `with`-block independence rule trades expressiveness for soundness

Â§9.7's "no sibling references inside a `with` block" is a parser-enforced, sound guarantee, but it is also strictly less expressive than Pulumi's or Beam's inferred-dependency graphs (Â§2.4) â€” a genuinely useful pattern like "three retrieval calls that could theoretically run in parallel, but the second wants to see the first's result to decide whether to bother" cannot be expressed inside one `with` block at all; it must be written sequentially, forfeiting the parallelism the author might have wanted. This is a deliberate trade (Â§9.7 states it explicitly), but it is a real expressiveness ceiling, not a solved problem.

## 24.3 The generics system is deliberately thin

Â§9.8's small, closed generic vocabulary (`Draft<T>`, `dataset<Row>`, `list<T>`) cannot express, e.g., a user-defined generic container over artifacts with its own merge semantics, or higher-kinded abstractions over "any capability that produces a T." A library author who needs this must drop to the stdlib's Rust implementation layer (Â§13.1, Â§15.17) rather than expressing it in Ulexite itself â€” an acceptable trade for a domain-specific language per Â§1.3, but a genuine limitation relative to a general-purpose language with a fuller type system.

## 24.4 Capability negotiation cannot fully close the provider-parity gap

Â§9.6's `structured_output: guaranteed | negotiated | unsupported` tiering is a real improvement over silent runtime failure (Â§3.4), but it can only reflect a capability difference the provider plugin author correctly declares â€” it cannot detect an *undeclared* gap (a provider that claims `guaranteed` but is subtly wrong under some input shape). This converts most of Â§2.3's "provider-agnostic, but leaky" problem into a compile-time-checkable one, but it depends on honest, well-tested plugin authors the same way any trait-based plugin system does (Â§12.4) â€” it is a mitigation, not an elimination, of the underlying problem.

## 24.5 Non-determinism typed at the call boundary, not eliminated

Â§9.3's `Draft<T>` makes non-determinism visible and exhaustively handled, but it does not make an LLM's output *correct* or *stable* â€” two runs with `cache: off` (Â§10.3) against the same prompt can still legitimately produce different `Settled(T)` values satisfying the same type. Ulexite's type system disciplines how a program *reacts* to non-determinism; it does not, and cannot, reduce the model's actual sampling variance. Judges (Â§5.6, Â§17.1) mitigate this at the evaluation layer, not the type layer, and judges themselves are probabilistic instruments requiring the calibration discipline in Â§17.1 to be trustworthy â€” a discipline the language encourages (Â§20.7's lint) but cannot force.

## 24.6 IR interpretation has a real (if usually dominated) performance ceiling

Â§13.6's choice to interpret rather than natively compile is justified by network-bound latency dominating interpretation overhead â€” but for a program with a very large, mostly-`Pure` computational core (heavy client-side artifact post-processing, large-scale embedding math done in-language rather than via a provider capability), interpretation overhead is no longer dominated by network latency and becomes a real, measurable cost. The mitigation is architectural, not eliminative: push genuinely heavy computation into a `python`/`shell` FFI call (Â§15.12) or a provider/tool plugin (Â§12.4/Â§12.6) written in Rust, rather than expressing it as Ulexite `Pure` IR nodes.

## 24.7 The declarative/imperative boundary is a design bet, not a proof

Â§4.5 and Â§10.2's split â€” declare the provably independent part, write the rest imperatively â€” is a bet that this two-region model covers the common case well enough that authors rarely fight it. It is possible real-world usage reveals a large class of programs that are "almost" parallelizable but don't fit `with`'s strict independence rule (Â§24.2) often enough that the ergonomic cost outweighs the soundness benefit; if so, a future revision (Â§25) may need a more permissive, effect-tracked parallelism model instead of the current syntactic one, at the cost of the simplicity Â§9.7 currently provides.

## 24.8 Tooling and ecosystem debt is real and unhidden

Â§22's package-ecosystem-maturity row honestly rates Ulexite "Low (new)" against LangChain/LlamaIndex's very large integration catalogues. Every provider, every vector store, every tool integration those ecosystems have accumulated over years must be either reimplemented as an Ulexite plugin or wrapped via FFI (Â§15.12) before parity is reached â€” Â§23.8 describes the wrapping path, but it is still work, not a solved problem, and some long tail of niche integrations may never justify a native port.

## 24.9 A new language imposes a real learning cost

Every team adopting Ulexite must learn a new grammar, a new type system's specific vocabulary (`Draft<T>`, `Verdict`, capability negotiation), and a new toolchain (Â§13, Â§20) â€” even with the migration paths in Â§23 designed to be incremental, this is a strictly higher up-front cost than adding one more Python import to an existing codebase, and that cost is only justified for teams whose conversation-orchestration surface is large/critical enough for the static guarantees (Â§9) to pay for themselves â€” Â§25 and Â§1.4 argue this bar is increasingly common, not that it is universal.

## 24.10 Real provider adapters cover most of the capability surface, with real gaps left

`crates/ulx-runtime/src/provider/` ships adapters for OpenAI-compatible servers (OpenAI, Groq, vLLM, LM Studio, and any other server speaking the same `/chat/completions` shape), Azure OpenAI, Anthropic, Gemini, Cohere, and Ollama, covering `chat` (all), `embed` (`openai_compatible`/`gemini`/`cohere`/`ollama`/`azure_openai`), `vision` (`openai_compatible`/`anthropic`/`gemini`/`ollama`/`azure_openai`), `judge` (all â€” every vendor's `"judge"` arm delegates to the shared `provider::judge` helper, which builds a rubric-evaluation prompt, sends it through that vendor's own `chat`, and parses the reply into a `Verdict`; Â§15.2's calibration discipline still applies since it's a real model call, not a deterministic check), and â€” `openai_compatible` only â€” `transcribe`/`speak`/`generate_image`. The gaps that remain are real, not hidden:

- **No artifact/blob system, by design choice.** Nothing in the runtime reads a file into a typed `Value` â€” a CLI `--arg doc=photo.jpg` is still a bare `Value::Text("photo.jpg")` (`crates/ulx-cli/src/main.rs`) that the *adapter itself* reads off disk at the HTTP-call boundary (`provider::artifact`), not something the interpreter or `Value` type knows about. `speak`/`generate_image` output is written to `$TMPDIR/ulexite-artifacts/<hash>.<ext>` and returned as a path string â€” a content-addressed filename, but not the pluggable `put(bytes)->hash`/`get(hash)->bytes` store Â§12.7 actually describes. A real artifact store remains future work. What `ulx run` *does* check today: before executing, `validate_run_args` (`crates/ulx-cli/src/main.rs`) looks up each `--arg`'s declared parameter type and, for `pdf`/`image`/`audio`/`video`, sniffs the referenced local file's magic bytes against that type via `ulx_runtime::artifact_validate` â€” a `doc: pdf` parameter fed a PNG is now rejected up front with a clear error instead of silently running. `http(s)://`/`data:` references and every non-binary artifact type (`text`/`markdown`/`json`/`xml`/`html`/`csv`/`embedding`/`vector`/`tool_output`) are still unchecked, since there's no fixed byte signature (or, for URLs, no local bytes) to sniff.
- **`vision` is images-plus-Anthropic-PDF, not full `ArtifactType::Pdf`/`Video` support.** jpg/png/gif/webp, read locally or passed through as an `http(s)://`/`data:` URI where the vendor's API fetches URLs directly (OpenAI-compatible, Azure OpenAI, Anthropic). Gemini and Ollama only accept a local file today (Gemini's URL-fetching path requires its separate File-API upload flow, not implemented; Ollama's native API has no URL-fetch concept at all). Anthropic additionally accepts a `.pdf` reference, routed to a `document` content block (`{"type": "document", "source": {"type": "base64"/"url", "media_type": "application/pdf", ...}}`, `provider::artifact::resolve_document`/`is_pdf_reference`) rather than the `image` block â€” every other vendor still rejects `.pdf` with a clear "PDF not supported for this vendor" error from `guess_image_mime`, and `ArtifactType::Video` is unimplemented everywhere. Both remain accepted by `ulx-sema`'s capability check (Â§9.6/Â§24.4) but fail at the HTTP boundary with a plain error for any unsupported vendor/artifact combination, not a type-checked rejection â€” the same "declared capability vs. what a plugin actually does" gap Â§24.4 already names.
- **`transcribe`/`speak`/`generate_image` are `openai_compatible`-only.** Anthropic and Cohere don't expose these APIs at all; Gemini, Azure OpenAI, and a native Ollama server do have their own equivalents, unimplemented here â€” Azure in particular offers its own Whisper/TTS/DALL-E deployments that `AzureOpenAiProvider` doesn't call. A user who wants these from another vendor is stuck with `openai_compatible` (OpenAI itself, or Groq for `transcribe`) until a future adapter is written.
- **`azure_openai` has no fixed default `base_url`.** Unlike every other hosted vendor, Azure's endpoint is per-customer (`https://<resource>.openai.azure.com`), so `base_url` is mandatory, as is the model field naming a *deployment*, not a generic model id â€” Azure resolves the underlying model server-side from that deployment, so the request body never repeats a `model` field the way OpenAI's does.
- **Retry/circuit-breaker is real but simple.** `provider::transport` now does exponential backoff with jitter (honoring a `Retry-After` header on 429) and a per-provider-instance circuit breaker that trips after repeated 5xx/transport failures â€” a real improvement over the single-fixed-retry this section used to describe, but still simpler than a production client's full policy (no per-error-class tuning, no distributed/shared breaker state across processes, a fixed retry-count ceiling).
- **Refusal detection is vendor-specific and not exhaustive.** Mapped to `Draft<T>`'s `Unsettled(Refused(..))` (Â§9.3) via each vendor's own finish/stop-reason field; Cohere's Chat v2 API exposes no such signal at all, so `CohereProvider` never produces `Refused`.

None of this is hidden behind a false "full parity" claim â€” it's the same honest-accounting posture as Â§24.4's capability-negotiation gap, just one layer further down the stack.

## 24.11 `.ulx`-declared providers and the stricter default have their own rough edges

Â§12.4's `provider` decl, `provider:`/`--provider` selection, and the new "no configured provider is an error, not a silent mock fallback" default (`crates/ulx-cli/src/providers.rs`, `crates/ulx-runtime/src/provider/mod.rs`) are real and tested, but not without gaps:

- **`judge`/`validator` calls can't carry a per-call `provider:` override.** Their args are consumed entirely into rubric *parameters* (`bind_rubric_params`, `interp.rs`) â€” there's no `Invocation.args`-equivalent bag the way an `ask` call has one. An ambiguous `"judge"` capability (two registered providers both serving it) can only be disambiguated globally, via `--provider` on the CLI, never at the `judge X(...)` call site itself.
- **`.ulx` provider capability params can't express a boolean.** `Expr` has no boolean literal anywhere in the grammar (a pre-existing gap, not new to this feature) â€” `chat: { model: "...", stream: true }` simply doesn't parse. `ulexite.toml`'s equivalent `params` table has no such restriction (TOML has real booleans). A capability param that happens to need one is only expressible from the manifest side today.
- **`ulx check`'s `from`/`provider:` reference validation is best-effort, gated on a manifest happening to be discoverable.** `pipeline::check` looks up `ulexite.toml` next to the file being checked the same way `ulx run` does â€” if it's not there, a `from "name"` or `provider: "name"` reference naming a raw manifest entry (rather than an in-language `provider` decl) can't be validated statically at all, and is silently deferred entirely to `ulx run`. This mirrors `check_ask_call`'s existing "silent when the check genuinely can't know" convention (Â§24.4), not a new kind of gap, but it means a clean `ulx check` is not a guarantee a subsequent `ulx run` won't fail on an unresolvable provider reference.
- **Eager, all-or-nothing provider construction.** Every declared provider (manifest entry or `.ulx` decl) visible to a run is built up front, regardless of whether the conversation actually being run ever asks for it â€” a broken `provider` block (e.g. a `from` with nowhere to inherit from) fails the whole run even if the specific conversation invoked never references it. This matches the manifest side's pre-existing behavior (already true before `.ulx`-declared providers existed), not a new regression.
- **A resumed run (`ulx approve`/`ulx deny`) re-resolves providers from scratch, but now defaults to the original `ulx run`'s `--mock`/`--provider` selection instead of requiring it again.** `RunManifest` persists `selected_providers`/`force_mock` (`crates/ulx-cli/src/manifest.rs`); `resume()` only falls back to them when the command itself was given neither flag, so an explicit `--provider`/`--mock` on `approve`/`deny` still overrides (e.g. deliberately migrating a run to a provider config that didn't exist yet when it started). `ulx replay` now reads the same persisted selection too, rather than always resolving unqualified.
- **`ulx run --interactive` prompts at the terminal for each `escalate(...)` instead of stopping for a separate `ulx approve`/`ulx deny`**, reusing the same probe-then-inject-into-cache replay `resume()` uses â€” it just loops in-process instead of across two CLI invocations. It always reports in plain text; `--output json`/`jsonl`/`mermaid`/`html` are ignored when `--interactive` is set. This is a synchronous convenience for a human at the terminal right now, not a replacement for the asynchronous suspend/checkpoint/resume design (Â§7.3, Â§10.7) â€” a human who isn't there yet still needs the real `ulx approve`/`ulx deny` flow.

## 24.12 Known compiler front-end gaps, found in a pre-v0.1.0 review

A code review ahead of tagging v0.1.0 surfaced several front-end (`ulx-syntax`/`ulx-sema`) gaps real enough to record here rather than silently leave undocumented â€” none block the fixes made alongside this review (runtime/CLI correctness, secret handling, provider bugs), but each is a genuine compiler-quality gap worth a fast-follow:

- **Unbounded parser recursion crashes the process on deeply nested input instead of producing a diagnostic.** Verified: a source file with tens of thousands of nested parens exhausts the recursive-descent parser's stack (an `mmap`/`mprotect` failure via `stacker`'s growable-stack allocator), aborting `ulx` outright rather than reporting a parse error. A malformed or adversarial `.ulx` file can crash the compiler.
- **Text-block interpolation splitting isn't brace/string-aware.** `{...}` interpolations are found by scanning for the first raw `}` byte, so a record literal or a string literal containing `}` inside an interpolation (both otherwise-valid `expr` forms) produces a spurious parse error instead of parsing correctly. The lexer's closing-`"""` scan has the same kind of gap in the other direction: it looks for the first three raw `"` bytes with no escape-awareness, so a `\"` inside the block immediately followed by two more literal `"` characters (rare, but possible) can be misread as the block's own closing delimiter.
- **`///` doc comments are discarded identically to `//`.** The grammar has a distinct `doc_comment` production (Â§8), but the lexer skips both forms the same way â€” no path in the compiler ever recovers doc text, so it's effectively dead syntax today.
- **Lexer errors report an imprecise 1-byte span**, and an overflowing integer literal (e.g. more digits than fit in an `i64`) is misreported as "unrecognized character" rather than a numeric-overflow-specific diagnostic.
- **Match-arm binding arity isn't validated statically.** `Fail => ...` (missing the `reason` binding) or extra bindings on a payload-less variant both type-check cleanly; a program referencing an unbound name only fails at runtime as "undefined variable," a case the compiler should catch statically.
- **`Verdict` exhaustiveness checking matches the literal type name, not a resolved type**, and general user-declared union / `Draft<T>` exhaustiveness checking (Â§9.3/Â§9.4) isn't implemented at all â€” only the specific name `"Verdict"` is special-cased. A user type literally named `Verdict` can defeat or confuse the check; any other union type gets no exhaustiveness checking whatsoever.
- **`retry`'s spec-mandated `else`-unless-provably-total rule (Â§8.1) is unenforced** â€” `retry(3) { ... }` with no `else` and a non-total body type-checks silently instead of being rejected.
