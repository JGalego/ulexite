# 3. Gap Analysis

## 3.1 Why the ecosystem still lacks a true language

Every system in §2 is, structurally, a **library loaded into a general-purpose host language**. That single fact explains nearly every weakness catalogued in §2.7–2.8:

- A library cannot add a keyword. It cannot make `judge` or `dataset` a grammar production with its own parsing and static-analysis rules — it can only offer a class, a decorator, or a YAML key, which is why Promptfoo's and OpenAI Evals' testing primitives dead-end in an escape hatch to raw Python/JS the moment logic gets non-trivial (§2.2).
- A library cannot add a type. It can define a Pydantic model or a `TypedDict` and ask the host language's type checker to enforce it, but the host checker doesn't know what a `Verdict` or an `Artifact<Image>` *means* — it can't reject "this step routes a video into a text-only model" or "this match doesn't handle the `Escalate` verdict" the way a language with those concepts baked into its type system can (§2.3, §2.7 point 2 and 4).
- A library cannot change the execution model. LangGraph, Temporal-for-workflows-style durability, and checkpointing all have to be adopted as an *opt-in* runtime wrapped around otherwise-ordinary host-language function calls — which is why every framework surveyed has a "the deterministic part isn't actually enforced, it's a convention" problem (§2.8) and why tracing/replay is either a paid SaaS product (LangSmith, LlamaTrace) or a DIY pattern (LlamaIndex's manual `Context` snapshotting).
- A library cannot make its own churn safe. Because the framework's classes and decorators *are* the program's surface syntax, every internal redesign (LangChain 0.1→1.0, LlamaIndex's three paradigm shifts, Semantic Kernel's planner removal and eventual full supersession) is necessarily a breaking change propagated straight into user code, with no stable IR standing between "how the framework happens to be implemented this quarter" and "what your program says." A language's grammar and IR are the thing that stays stable while the standard library and runtime evolve underneath it (§13.4).

The pattern is not "nobody has tried hard enough." DSPy, Guidance, and LMQL specifically tried to compile something (a tuned prompt, a token grammar, a masked decode) — real compiler-shaped effort exists in this space. But each compiles to an artifact coupled to one execution substrate (one model, one backend tier, one dataset), not a portable IR a runtime can retarget across providers the way LLVM IR retargets across CPUs. That is the gap: **partial compilers exist; a full language — grammar through IR through a provider-independent runtime — does not.**

## 3.2 What people are forced to write by hand today

Every item below is corroborated by a specific finding in §2, not a general impression:

| Concern | What people hand-write today | Evidence |
|---|---|---|
| **Prompt orchestration** | String concatenation/templating (Jinja/f-strings) with no static check that a variable exists or a type matches | LangChain: missing template variable is a runtime `KeyError`, never a compile error (§2.3) |
| **Conversation state** | A list of messages threaded manually through every call, or wrapped in a session/history object the developer must remember to read-before/write-after | LangChain's `RunnableWithMessageHistory` + stringly-typed `session_id` config key; DSPy's manually-appended `dspy.History`; OpenAI Agents SDK's three coexisting mechanisms (manual list, Session, server-managed) (§2.1, §2.3) |
| **Retries** | Ad hoc `try/except` loops, or a wrapper object (`.with_retry()`, `RetryPolicy`) whose actual backoff/ordering semantics are opaque until you read the wrapper's source | LangChain's `.with_retry()`/`.with_fallbacks()`; LangGraph's `RetryPolicy` with a documented case of retries silently not respected (langgraph#6027) (§2.3) |
| **Tracing** | Either a paid third-party SaaS bolted on via callback instrumentation, or nothing | LangSmith (LangChain), LlamaTrace/Phoenix (LlamaIndex) — both external products, not core-language features (§2.3) |
| **Evaluations/benchmarks/judges** | A separate config file (YAML) or product, disconnected from the orchestration code's types, escaping to raw code for anything non-trivial | Promptfoo, OpenAI Evals (§2.2); LlamaIndex's `CorrectnessEvaluator` as an async batch job, not an inline primitive (§2.3) |
| **Datasets** | JSONL/CSV files with no schema link to the code that consumes them, no versioning, no fixture-style reuse | OpenAI Evals' `{"input":..., "ideal":...}` JSONL with no static link to the completion function's actual signature (§2.2) |
| **Provider switching** | A per-framework adapter layer, with quietly degraded feature parity when switching (structured output, multimodal input, tool calling often silently unsupported on non-primary providers) | OpenAI Agents SDK's LiteLLM integration explicitly documented to lose multimodal input and structured-output reliability on non-OpenAI backends (§2.3) |
| **Multimodal routing** | Ad hoc per-content-type handling, with no unifying artifact type the compiler can check | LlamaIndex's `ImageBlock`s and LangChain's new (2025) standardized content blocks are steps forward but remain library-level Pydantic types, not compiler-checked artifact types (§2.3, §2.5) |

## 3.3 Why these should become language primitives, not library features

The argument is the same in every row of §3.2, and it is the same argument that justified SQL, Terraform, and Playwright each becoming a language rather than remaining a library:

1. **Static guarantees require the compiler to know the domain's nouns.** A general-purpose type checker can verify `dict[str, Any]` is well-formed; it cannot verify "this conversation never routes an `audio` artifact into a model that only accepts `text`" unless *artifact* and *model capability* are types the compiler itself understands (§9). That requires the domain's nouns — conversation, message, artifact, judge, verdict — to be language keywords with grammar productions, not library classes.
2. **Cross-cutting runtime guarantees (retry, cache, trace, replay) need to be the default, not a wrapper someone remembered to apply.** Every framework in §2.3 makes tracing and retries an opt-in decoration around an otherwise bare call; every incident report in §2.8 traces back to a call site where the decoration was missing, silently misconfigured, or itself buggy. A language can make "every model call is traced and content-addressed cached" the *base case*, the way `git commit` always produces a hashed object — there is no code path that skips it.
3. **Testing primitives need to share the host program's types to be worth anything.** Promptfoo/OpenAI Evals' YAML tests can assert `is-json` against a string; they cannot assert "the judge's verdict matches the `Verdict` enum the orchestration code itself pattern-matches over," because the test config and the orchestration code have no shared type system. Making `expect`/`judge`/`dataset` grammar means a test and the program it tests are checked by the same compiler pass.
4. **Provider independence needs to be a compiler/runtime contract, not an adapter each framework author maintains by hand.** Every "provider-agnostic" layer surveyed (LiteLLM in DSPy and OpenAI Agents SDK, LangChain's `init_chat_model`, Semantic Kernel's connectors) is a best-effort adapter that leaks capability differences at runtime. A language can instead require every provider plugin to declare its capabilities (§12.4) and have the compiler check a program's requirements against that declaration before a single token is spent — the same shift SQL made from "hope the storage engine supports this query shape" to "the planner knows what the engine can do."
5. **Churn-proofing requires an IR the framework's internal rewrites don't reach.** §2.8's version-churn catalogue (LangChain, LlamaIndex, Semantic Kernel) is not evidence that framework authors are careless — it's evidence that, absent a stable IR, *every* internal improvement is externally breaking. A real compiler pipeline (source → AST → IR → runtime, §13) lets the runtime and stdlib evolve underneath a stable grammar the way libc can change underneath a stable C ABI.

## 3.4 What Ulexite must not inherit

To be more than "LangGraph with different syntax," Ulexite's design must resolve, not merely relabel, the specific failures documented above:

- Reducers (LangGraph) become the language's default artifact-merge semantics (§9.5), not an opt-in `Annotated[...]` field a user must remember to add — merge conflicts are a compile-time-checkable property of two steps writing the same artifact.
- Checkpointing (LangGraph/Temporal) becomes unconditional — every conversation run is checkpointed, the way every git commit is hashed — rather than a `durability=` parameter a caller can silently leave at the crash-prone default (§10.4, §12.5).
- Judges and validators (Promptfoo/OpenAI Evals/LlamaIndex) share the exact `Verdict` type used by ordinary control flow (§5.6–5.7, §9.4), so a test suite and the program under test are checked by the same compiler pass, closing the gap in §3.3 point 3.
- Provider capability negotiation (LiteLLM/`init_chat_model`/SK connectors) is a compile-time-checked capability declaration (§12.4), not a runtime adapter that silently drops multimodal or structured-output support.
- The stdlib (§15), not the grammar, absorbs technique that will keep evolving — DSPy-style prompt optimization, judge rubrics, RAG retrieval primitives — so that the kind of improvement that forced LangChain/LlamaIndex/SK's repeated breaking rewrites lands as a versioned library upgrade against a stable IR, not a language-level break.
