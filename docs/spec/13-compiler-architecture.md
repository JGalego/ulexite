# 13. Compiler Architecture

## 13.1 Implementation language: Rust

**Decision: Rust**, for the reference compiler, runtime, and CLI (`ulx`). Evaluated alternatives and why each loses:

| Language | Verdict | Reasoning |
|---|---|---|
| **Rust** | **Chosen** | No-GC memory safety for a long-running runtime process juggling many concurrent, cancellable, checkpointed conversations; a mature async ecosystem (tokio) suited to §12.3's I/O-bound concurrent provider calls; a first-class WASM target (browser-based playground, §20.10, sandboxed tool execution); a real precedent in this exact domain — Guidance's own token-constrained decoding engine (`llguidance`) migrated from Python to Rust specifically for the performance and correctness §2.1 documents; and a package/tooling culture (`cargo`, `crates.io`, `rust-analyzer`) that is itself a design model for §14 and §20. |
| Go | Rejected | GC pauses are a poor fit for a runtime whose whole value proposition is deterministic, precisely-timed replay (§10.4); weaker sum-type/pattern-matching support than Rust makes encoding `Draft<T>`/`Verdict` exhaustiveness (§9.3–9.4) in the compiler's own implementation more error-prone, ironic for a language whose selling point *is* that exhaustiveness. |
| OCaml | Rejected on ecosystem grounds, not technical merit | Excellent fit technically (ADTs, pattern matching, a real compiler-writer's language — this is what Rust borrowed the ergonomics from) but a materially smaller talent pool and library ecosystem for the surrounding needs (async runtime maturity, WASM target maturity, LSP tooling libraries) than Rust's, and the mission's "prefer mature, widely adopted ecosystems" tips the balance against it despite OCaml being, on paper, the more natural compiler-writer's choice. |
| F# / C# | Rejected | Strong ADT/pattern-matching story via F#, but both are CLR-tied; a CLR runtime dependency for the reference implementation of a systems-adjacent tool cuts against "avoid fragile or trend-driven dependencies" and against the WASM/embeddability goals above. |
| Zig | Rejected for the compiler; philosophy adopted for the language design | Zig's no-hidden-control-flow ethos directly inspired §4.7/§9.3's explicit error-union design (§2.5) — but Zig in 2026 is still pre-1.0, with an evolving standard library and language spec, which fails the mission's "mature, stable" bar for the thing actually shipping the reference implementation, however aligned its philosophy is with Ulexite's own. |

Python is explicitly out of scope as the implementation's architectural foundation per the mission brief; it appears only as an FFI target (§9.1, §15.12) and for optional scripting bindings, never as the compiler or runtime's own substrate — precisely to avoid becoming "a Python framework with new syntax painted on," the failure mode §2 spends most of its length documenting.

## 13.2 Parser

**Lexer:** `logos` — a derive-macro-based, DFA-compiled lexer generator, mature and widely used in production Rust compilers/tools, chosen over hand-written regex scanning per the mission's explicit "avoid ad hoc regex parsing" requirement.

**Parser:** `chumsky` — a parser-combinator library with first-class error recovery (parses past a syntax error to report multiple diagnostics in one pass, essential for §20's language-server experience) and no code-generation step (unlike LALR generators), making the grammar in §8 directly executable as combinator code reviewable against the EBNF one production at a time. Chosen over a generated-parser approach (e.g. LALRPOP) because chumsky's error-recovery story is significantly better suited to an IDE-first tool (§20) where a program is syntactically invalid for most of its editing lifetime — the same reason production language servers increasingly favor recovery-capable combinator/PEG parsers over strict LALR.

The grammar in §8 is the source of truth; the `chumsky` parser is generated by hand from it (not auto-derived), with a golden-file test suite (one `.ulx` fixture per grammar production) verifying the parser accepts/rejects exactly what §8 specifies — the parser and the formal grammar are two artifacts checked against each other in CI, not one artifact treated as self-evidently correct.

## 13.3 Pipeline stages

```
source (.ulx)
   │  lexer (logos)
   ▼
tokens
   │  parser (chumsky)
   ▼
AST (concrete, with source spans on every node)
   │  name resolution + import resolution (§14)
   ▼
resolved AST
   │  semantic analysis:
   │    - artifact-type checking (§9.2, §11.5)
   │    - Draft<T>/Verdict exhaustiveness checking (§9.3-9.4)
   │    - with-block independence checking (§9.7)
   │    - merge-function requirement checking (§9.5)
   │    - capability negotiation checking (§9.6)
   ▼
typed AST
   │  lowering
   ▼
IR (§13.4)
   │  optimizer (§13.5)
   ▼
optimized IR
   │  (interpreted directly by the execution engine, §12.2 — no native codegen; see §13.6)
```

Each stage is a separate crate with its own unit tests (parser tests, semantic-analysis tests per diagnostic category, IR-construction tests) per the mission's testing requirements (§25).

## 13.4 Intermediate representation

The IR is a typed, SSA-like graph of **effects** and **pure operations**, deliberately the artifact the frameworks in §2.1 each almost-built and then over-coupled to one substrate (a token grammar, a masked decode, a tuned prompt):

- **Pure nodes**: variable bindings, `match` arms, artifact field access, merge-function application — evaluated identically on every run, replayable by re-execution rather than memoization.
- **Effect nodes**: capability calls, tool calls, human-approval waits — each tagged with its declared capability requirement (§9.6) and its position in the two-pass schedule (§10.2), memoized during replay (§10.4) rather than re-executed.

This pure/effect distinction is exactly what makes §10.4's replay guarantee checkable at compile time rather than assumed: an IR node is one or the other, never ambiguously both, and the checker in §13.3 that verifies `with`-block independence (§9.7) is a direct consumer of this distinction. The IR is provider-independent by construction — it references a *capability*, never a vendor — satisfying the mission's "the parser and compiler should be independent from any LLM provider" requirement structurally, not by convention.

## 13.5 Optimizer

Optimization passes operate only on the declarative region (§10.2) and are conservative by design, since the domain's real cost (LLM call latency/price) dwarfs any interpretation overhead the optimizer could remove:

- **Common subexpression elimination on effect nodes** — two `with`-block bindings issuing byte-identical capability calls (same capability, same resolved provider, same input artifact hashes, same parameters) are merged into one effect node before scheduling, complementing (not replacing) the runtime cache in §10.3 — this catches the case at compile time instead of paying even a cache-lookup cost at run time.
- **Dead-artifact elimination** — a binding never referenced downstream and not the subject of a `snapshot`/`expect` is not scheduled at all, avoiding spending tokens on output nothing consumes (a static-analysis warning surfaces this to the author too, §20.7).
- **Independence-preserving reordering** — within a `with` block, bindings may be reordered for scheduling locality without changing program meaning, since §9.7 already guarantees no ordering dependency exists.

No optimization pass is permitted to reorder or merge anything in the imperative region (§10.2) — the same principle as §4.5's syntax-level boundary, enforced again at the IR level so an "optimization" can never silently turn a sequential, decision-dependent conversation into a differently-ordered one.

## 13.6 Execution strategy: IR interpretation, not native codegen

The optimized IR is interpreted directly by the execution engine (§12.2), not compiled to native machine code. This is a deliberate choice, not an oversight: the domain's bottleneck is network-bound LLM/tool latency (hundreds of milliseconds to tens of seconds per effect node), several orders of magnitude larger than any plausible interpretation overhead — the same reasoning that lets SQL engines, Terraform, and Airflow all interpret a plan rather than JIT/AOT-compile it. Interpretation also keeps §10.4's replay guarantee simple: replaying a checkpoint log means re-interpreting the same IR against recorded effect results, with no native-code-versioning concerns (a replayed run compiled by a different compiler version reinterprets the same portable IR, rather than needing bit-identical native code).

## 13.7 Incremental compilation

Because the AST and IR carry source spans and a stable node-identity scheme (content hash of the node's own syntax, independent of unrelated edits elsewhere in the file — a direct analogue of §11's artifact content-addressing, applied to compilation units instead of runtime artifacts), re-compiling a file after a small edit reuses unaffected AST/IR subtrees rather than rebuilding the whole file's IR, the mechanism behind §20.7's sub-second incremental language-server diagnostics and behind incremental `ulx build` for large multi-file packages (§14).
