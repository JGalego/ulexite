# 2. Prior Art Survey

This survey does not summarize each system's marketing. It extracts, per system, the load-bearing abstraction, the execution model, and — most importantly — the accidental complexity and missing primitives that a language designer needs to either fix or deliberately not repeat. Findings are grounded in current (2025–2026) documentation, source, and issue trackers, not stale training-data impressions; version-specific claims are dated.

## 2.1 Constrained-generation DSLs: Guidance, LMQL, DSPy

**Microsoft Guidance.** The unit of composition is a Python function (`@guidance`) building an immutable `lm` object via `+=`, with `gen()`/`select()` primitives and role context managers. Since v0.2.0 (2024), constraint enforcement is delegated to `llguidance`, a Rust engine doing real per-token grammar-constrained decoding (Earley parsing + regex-derivative lexing) in ~50μs/token — a genuinely strong result. But the guarantee is backend-tiered: full token-masking requires local logit access (llama.cpp, Transformers, vLLM, SGLang); remote APIs (OpenAI, Azure) fall back to "optimistic running" — approximate constraints validated post hoc. Guidance proves token-level structural guarantees are achievable and fast, but ties the guarantee to a backend capability tier rather than making it a portable language contract. It has no retry/eval harness, no cross-provider abstraction (each backend is a distinct model class), thin and backend-conditional multimodal support, and has undergone at least two backward-incompatible rewrites (v0.1, then the v0.2 grammar-engine swap).

**LMQL.** Decoder clauses (`argmax`/`sample`/`beam`) plus a `where` constraint clause translate to eager per-token logit masking, requiring logprob/token-level backend access — the same architectural coupling as Guidance, and it degrades the same way on chat-only APIs (OpenAI's Chat Completions API supports only a subset of constraints, and its 300-logit-bias cap forces LMQL to silently truncate masks). LMQL is the clearest evidence in this survey that **constrained decoding, as an approach, is architecturally coupled to a specific inference-serving capability** — a lesson Ulexite must design around by making structured-output validation a runtime-negotiated capability (§9.6, §12.4), not a hardcoded assumption. LMQL shows no built-in retry/backoff, no first-class multi-turn state object, no multimodal support, and — as of mid-2025 — near-zero commit activity, a cautionary data point about the maintenance burden of owning a custom token-level decoding runtime per backend.

**DSPy.** The most conceptually mature of the three: a `Signature` (`"question -> answer"` or a typed class) declares I/O shape independent of prompt phrasing; a `Module` (`Predict`, `ChainOfThought`, `ReAct`) wraps a signature with a strategy; an optimizer ("compiler") searches over instructions/few-shot demonstrations against a metric on a devset. This is the strongest prior art for "separate what a step should produce from how the model is coaxed into producing it" — but DSPy's compilation targets *prompt content*, not an executable IR, so "compiling" produces a prompt artifact tied to a specific model/dataset/optimizer configuration that can silently degrade as the underlying model drifts and typically must be recompiled. Its built-in eval harness (`dspy.Evaluate`) and retry/self-correction (`dspy.Refine`/`dspy.BestofN`, replacing deprecated `Assert`/`Suggest` in 2.6) are real, but multi-turn conversation (`dspy.History`) is confirmed bolted-on: appended manually per turn, with asymmetric few-shot rendering. Tracing is not native; it's via an MLflow integration.

**Cross-cutting lesson:** all three prove that a compiler-like transformation over LLM interaction (grammar compilation, query compilation, prompt-optimization compilation) is both possible and valuable — but each compiles to a different, backend- or model-specific artifact (a token grammar, a masked decode, a tuned prompt) rather than a portable IR a runtime can execute against *any* provider. Ulexite's IR (§13.4) must be the thing these three each independently almost-built and then coupled too tightly to one execution substrate.

## 2.2 Evaluation and testing frameworks: Promptfoo, OpenAI Evals

**Promptfoo.** A YAML matrix (`prompts × providers × tests`) graded per cell by typed assertions — deterministic (`regex`, `json-schema`) and model-graded (`llm-rubric`), combined via weighted scoring against a threshold, with content-addressed disk caching by default. This is genuinely good prior art for §5.8's testing primitives: matrix testing and a built-in judge type belong in the language, not a config file. Its limits are structural to YAML-as-host: control flow lives in Nunjucks-template hacks or an escape hatch to JS/Python the moment logic is non-trivial; there is no fixture/parametrize/plugin system comparable to pytest; and — critically for a conversation-first language — conversation history is "just another var," so Promptfoo evaluates a final output, never a trajectory.

**OpenAI Evals.** The cleanest abstraction in this cluster: three independent concerns — dataset (JSONL), completion function (how an answer is produced), and eval class (how it's graded) — composed via a registry. This three-way separation is worth adopting nearly as-is (§16.4). But the YAML registry is only a routing table; any real eval logic requires a Python subclass, so "config-only" breaks down immediately, exactly like Promptfoo. Crash recovery is eval-set-granularity, not mid-eval. The platform's hosted Evals API is itself being sunset (November 30, 2026) — direct evidence that config/product-level eval tooling built outside a language has a short half-life compared to a language primitive.

**Cross-cutting lesson:** both systems prove the industry wants matrix testing, LLM-as-judge grading, and golden datasets badly enough to build entire products around a YAML file — and both hit the same wall at the same point: real logic escapes the config format into ad hoc code with no shared type system, fixture model, or IDE support. This is precisely the gap §16 closes by making `dataset`, `judge`, and `expect` grammar, not schema.

## 2.3 Agent orchestration frameworks

**LangGraph.** `StateGraph` + `Annotated[type, reducer]` fields is the standout idea here: reducers give an explicit, typed answer to "how do concurrent branches merge into shared state" (§9.5 borrows this directly for artifact merges). `MessagesState`/`add_messages` gives a first-class typed message-history field, and checkpointing — a `(thread_id, checkpoint_id)`-addressed, replayable, forkable snapshot after every superstep, with an explicit `durability="exit"|"async"|"sync"` consistency knob — is the closest any surveyed system comes to §5.9's runtime guarantees. Its weaknesses are concrete and documented, not speculative: Pydantic state-schema validation has open, confirmed bugs where validation failures produce stack traces with no indication of which node failed (langgraph#1977, #1978, #4060); a v1.0.2 patch release broke a subclassed interface days after a "no breaking changes" promise (#6363); and there is no compile-time verification of state/tool schemas — everything is checked, partially, at runtime. There is also no built-in judge/eval primitive (evals live in the separate LangSmith product) and no typed multimodal artifact type.

**LangChain.** The `Runnable` protocol (uniform `invoke/batch/stream` + async twins across every component) composed via the `|` operator is a genuinely good idea — a single interface a conversation, a turn, and a tool call could all satisfy (§10.1). But LangChain is also the single best-documented cautionary tale in this survey for **framework churn as chronic condition**: v0.1 → v0.2 → v0.3 → v1.0 (October 2025) each broke imports; the 1.0 release stripped the core package down to four concepts and exiled everything else (including LCEL's own former center of gravity) to `langchain-classic`. Practitioner postmortems are blunt and specific: a production team's 12-month retrospective cites "five layers of abstraction just to change a minute detail" and no way to externally observe an agent's internal state; a widely-read critique describes chains/parsers/callbacks stacking into "a dependency graph that looks like spaghetti code." Conversation history is a plain list of typed `BaseMessage` objects wired through an external `RunnableWithMessageHistory` wrapper and a stringly-typed `session_id` config key — better than raw dicts, still not a language-level object. No compile-time checking exists anywhere in the pipeline: a missing template variable is a runtime `KeyError`.

**Semantic Kernel.** `Filters` — composable `next(context)` middleware around function invocation and prompt rendering — is a clean, directly reusable idea for wrapping every turn (§12.6). `ChatHistory` as a typed, serializable object with multimodal content items is solid. But SK is the survey's clearest example of a framework designing itself into obsolescence: its own **planner subsystem was deprecated and removed wholesale**, replaced by model-native function-calling; a GitHub feature request (#13435) asks SK to add exactly what Ulexite treats as day-one primitives — step-level checkpoints, deterministic replay without re-triggering side effects, side-effect classification per tool for safe retry — none of which exist in core SK today; and as of October 2025 the *entire framework* is in maintenance mode, superseded by Microsoft Agent Framework, whose own migration guide states plainly "there's no longer a Kernel object to use." A framework's central abstraction being retired by its own successor, twice in three years, is strong evidence that "the orchestrator object" is the wrong place to put the language's center of gravity.

**OpenAI Agents SDK.** Deliberately minimal (`Agent`, `Tool`, `Handoff`, `Guardrail`, `Runner`, `Session`), with tracing on by default and free — every run auto-generates a trace of typed spans, exported to 27+ external backends. Guardrails run concurrently with the model call by default, a good latency-conscious default. Its real gaps are precise: Human-in-the-loop was a *bolted-on afterthought* forced by community pressure (issue #636, 38 comments, "should be implemented on top priority"); checkpointing is limited to pending-tool-approval granularity via `result.to_state()`, not arbitrary mid-workflow snapshots; and "provider-agnostic" via LiteLLM is real but leaky by the docs' own admission — structured outputs, multimodal input, and hosted file search are OpenAI-only, and other providers "occasionally produce invalid JSON." No compile-time checking of tool schemas or handoff-input-type compatibility exists; mismatches surface at runtime as `ModelBehaviorError`.

**LlamaIndex.** `Workflow` infers a step's dependency graph from typed event signatures (`@step async def f(ev: TypeA) -> TypeB | TypeC`) rather than requiring an explicit graph object — a genuinely lighter-weight alternative to LangGraph's explicit `StateGraph` worth partially adopting for Ulexite's step-to-step wiring (§10.2). `Context.to_dict()/from_dict()` making full in-flight execution state a serializable value is a good pause/resume primitive. But LlamaIndex has undergone three stacked paradigm shifts in its history (`ServiceContext` → global `Settings`; `QueryPipeline` → `Workflow`; `AgentRunner` → `AgentWorkflow`), leaving fossilized tutorials and, by the project's own docs, workflows are explicitly "ephemeral by default" — durable execution is a DIY pattern of manually snapshotting to Redis, not a runtime guarantee. Chat history and workflow state are two separate, unreconciled mechanisms (`ChatMemoryBuffer` vs. `Context`), and no compile-time check verifies a workflow's event graph is even connected — a static type-checker will happily accept a `Workflow` that can never reach `StopEvent`.

**Cross-cutting lesson:** every orchestration framework surveyed independently reinvented (a) a typed-ish message/history object, (b) some form of checkpoint/session persistence, and (c) some retry/guardrail middleware — and every one of them treats these as runtime, library-level concerns bolted onto a general-purpose host language, checked (if at all) at runtime, not compile time. None has a static type system that can reject a program routing the wrong artifact type into a step, or forgetting to handle a verdict variant, before the program runs once. That gap — not orchestration topology, which these frameworks have mostly solved well — is Ulexite's opening.

## 2.4 Infrastructure-as-code and pipeline systems

These systems were not built for LLMs, but each solved a scheduling/execution-model problem Ulexite also has:

- **Terraform** — declarative resource graph, `plan`/`apply` split, state file as reconciliation target. Maps to previewing a conversation's token cost/side effects before running it (§10.5). Weakness to avoid: HCL is a config language wearing a programming-language costume — loops (`count`/`for_each`) are bolted on, not native.
- **Pulumi** — proves you don't need a bespoke DSL; a host language's SDK can build the same resource graph via ordinary function calls. Weakness: because graph construction is arbitrary imperative code, "preview" is not guaranteed pure — Ulexite avoids this by making the effectful model/tool call itself, not graph construction, the one place non-determinism is allowed to enter (§4.4).
- **GitHub Actions** — event-triggered DAG of jobs/steps with a `matrix` fan-out. Maps to parallel exploration over prompts/models/params. Weakness: YAML has no real composability or local test story.
- **Bazel** — hermetic, content-addressed build graph; caching keyed on hashed inputs. Maps directly to caching LLM calls by `hash(prompt + model + params + context)` (§4.5, §12.5). Weakness: steep learning curve, painful incremental adoption.
- **Temporal** — durable execution: deterministic workflow code, persisted event history, replay reconstructs state; non-deterministic work is isolated into "activities" with automatic retry and replay-time memoization. This is the single closest architectural match in the entire survey to §4.4/§10.4's reproducible-conversation-replay model — LLM/tool calls are Ulexite's "activities." Weakness to avoid: Temporal's determinism constraint on workflow code is a convention the programmer must not violate, with no compiler enforcement; Ulexite makes this a type-system distinction instead (§9.3) so it cannot be violated by accident.
- **Apache Beam** — `PCollection`/`PTransform` decouple pipeline definition from execution engine ("runner"). Maps to decoupling a conversation definition from its execution backend. Weakness: the abstraction leaks in practice — performance tuning ends up runner-specific anyway.
- **Nextflow** — dataflow/channels give implicit parallelism purely from data availability, plus content-hash task caching (`-resume`) for pipeline resumability. Maps to automatic parallel fan-out of independent conversation steps. Weakness: channel consumption semantics are a frequent source of subtle bugs.
- **Airflow** — DAG of tasks, scheduler-driven retries/backfill, "sensors" for waiting on external events. Maps to human-approval-as-sensor. Weakness, directly relevant: the historical requirement that a DAG be statically known at parse time makes genuinely dynamic, data-conditioned workflows awkward — exactly the failure mode Ulexite avoids by keeping turn-by-turn control flow imperative (§4.5) while only the *provably* independent sub-graph is planned declaratively.

## 2.5 Language and type-system inspirations

- **SQL** — separates declarative *what* (a query over relations) from the *how* (a cost-based physical plan); CTEs (`WITH x AS (...)`) name intermediate results composed by reference, free to be reordered/parallelized/cached independently. This is the direct model for Ulexite's declarative sub-block (§7.2).
- **GraphQL** — a typed schema plus a client-specified selection set, so response shape is validated *before* a resolver runs. Maps to requesting an exact typed shape of structured output from a model rather than "call and hope the JSON matches" (§9.2, §15.1).
- **React** — declarative component tree, unidirectional data flow, and a reconciler that diffs old/new trees to compute minimal re-execution. Maps to conversation steps as memoized components that only re-run when their declared upstream artifacts actually change (§11.3).
- **Rust** — algebraic data types with exhaustive `match`, and `Result<T, E>` instead of exceptions. Maps directly to `Verdict` (§9.4, §5.6) and to model calls returning a typed union rather than throwing.
- **Zig** — no hidden control flow, explicit error-union return types handled via `try`/`catch`. Maps to making retry/timeout/degraded-mode outcomes visible in a call's type signature (§9.3), not ambient framework behavior.
- **Gleam** — sound static typing over BEAM actors; a process's mailbox has a statically known, closed message type, so an unhandled message shape is a compile error. Maps to closed, exhaustively-matched participant/message-role types (§5.2).
- **Elixir** — supervision trees and "let it crash": a process is allowed to fail; a supervisor restarts it per a declared strategy, bounding the blast radius. Maps directly to Ulexite's `supervise` construct for steps (§12.6), replacing the try/except-per-call pattern seen in every framework in §2.3.

## 2.6 Testing/automation tool ergonomics

- **Playwright** — auto-waiting (actionability checks baked into every action) and web-first assertions that poll until a condition holds or a timeout expires, instead of checking once, map directly onto polling an LLM judge to convergence or a tool call to a terminal state (§16.3). Its trace viewer (scrubbable timeline of every step) is the direct model for Ulexite's trace viewer (§18, §20). Its visual snapshot testing is the model for golden-output testing — with the necessary divergence that the diff must be semantic, not pixel- or line-based, since exact-match text is far too brittle for non-deterministic output (§16.5).
- **git** — the content-addressed, immutable object model (blobs/trees/commits keyed by hash) plus branches-as-pointers is the direct model for Ulexite's artifact/trace storage (§11.2, §18.2): every message and artifact is hashed and deduplicated, a conversation is a chain of immutable snapshots, and forking a conversation at any turn to try an alternate prompt is cheap. The one thing that does *not* transfer is line-based diff/merge — semantic content needs a semantic diff, not git's algorithm, even though the storage model is reused as-is.
- **pytest** — fixtures as dependency-injected functions with explicit scope, `@parametrize` turning one test into many independently reported cases, and assertion rewriting for readable diffs, all as real language/runtime features rather than an external config schema, is the direct model for making `expect`/`assert`/`dataset`/`benchmark` grammar (§16) rather than a decorator-based add-on the way Promptfoo/OpenAI Evals remain.

## 2.7 What recurs everywhere

Across all sixteen systems, four abstractions were independently reinvented, in incompatible and partial forms, by nearly everyone:

1. **A message-history object** (Guidance's `lm`, LangGraph's `MessagesState`, LangChain's `BaseChatMessageHistory`, SK's `ChatHistory`, OpenAI Agents SDK's `Session`, LlamaIndex's `ChatMemoryBuffer`) — always bespoke, always library-level, never checked against a static schema.
2. **A checkpoint/replay mechanism** (LangGraph's checkpointer, LlamaIndex's `Context.to_dict()`, OpenAI Agents SDK's tool-approval state, Temporal's event history) — ranging from "real and load-bearing" (LangGraph, Temporal) to "acknowledged wishlist item" (Semantic Kernel's own open feature request) to "DIY pattern" (LlamaIndex).
3. **A retry/guardrail middleware layer** (SK's Filters, LangGraph's `RetryPolicy`, OpenAI Agents SDK's Guardrails, LangChain's `.with_retry()`/`.with_fallbacks()`) — always bolted on as a wrapper around an otherwise unchecked call, never a type the compiler can verify was handled.
4. **A judge/eval mechanism** (Promptfoo's `llm-rubric`, OpenAI Evals' `ModelBasedClassify`, LlamaIndex's `CorrectnessEvaluator`) — always a separate product or module bolted onto the orchestration codebase, never able to see the orchestration language's own types.

## 2.8 Accidental complexity to explicitly not repeat

- **Version churn as chronic condition**, not a one-time migration: LangChain (four major breaks in 21 months), LlamaIndex (three stacked paradigm shifts), Semantic Kernel (repeatedly rewritten, now in maintenance mode, superseded by its own successor). Root cause common to all three: the "framework" *is* the language surface, so every internal refactor is a breaking change to every user's program. Ulexite's answer is a stable grammar and IR (§8, §13.4) with library-level stdlib modules (§15) versioned independently and swappable without touching program syntax.
- **Runtime-only checking of everything**: no system surveyed statically verifies artifact-type compatibility, exhaustive verdict handling, or prompt/schema consistency before first execution. This is the single most consistent, most fixable gap (§9, §20.7).
- **Config-language-as-fake-programming-language**: Promptfoo/OpenAI Evals' YAML, Terraform's HCL, GitHub Actions' YAML all hit the same wall the moment real control flow is needed, and all solve it by escaping to embedded Python/JS/shell with no shared type system. Ulexite's answer is to never draw that line: control flow is native grammar from the start (§4.5, §6.4).
- **Determinism-by-convention instead of by construction**: Temporal's replay-determinism rule and Pulumi's "preview should be pure" both depend on the programmer not violating an unenforced convention. Ulexite encodes the non-deterministic boundary in the type system instead (§9.3).

## 2.9 Where existing systems are genuinely superior (and Ulexite should not pretend otherwise)

- LangGraph's checkpoint/thread/durability model is more battle-tested in production, at more scale, than anything Ulexite can claim on day one; §22 and §24 say so plainly.
- DSPy's optimizer research (MIPROv2, GEPA) represents real, hard-won empirical technique for automatic prompt improvement that Ulexite's stdlib should wrap (§15.14), not reinvent from scratch.
- OpenAI Agents SDK's tracing-on-by-default, exported to 27+ backends, has an ecosystem integration breadth a new language starts with zero of.
- Guidance/LMQL's token-level grammar constraint enforcement, where backend logit access is available, is a stronger guarantee than anything a provider-agnostic language can promise uniformly (§9.6 is explicit that this degrades gracefully, not silently, across providers).
